# JSONL v2.1+ 兼容性更新设计

**日期：** 2026-05-13
**范围：** parser (cc-session-jsonl) + 分析层 (cc-token-usage) + Vue 前端
**版本目标：** cc-session-jsonl 0.2.0，cc-token-usage 2.1.0

## 背景

Claude Code 2.0.77+ 在 JSONL 数据上做了两类不兼容增量：

1. **新字段**：`user` / `assistant` entry 上新增了 attribution、tool-result 关联、permission-mode 等字段。当前 parser 用 serde 静默丢弃。
2. **新项目级元数据文件**：`<project>/sessions-index.json`、`<project>/bridge-pointer.json`、`<project>/<uuid>/tool-results/<id>.txt`。当前 scanner 完全无感。

23 种 entry type tag 全部覆盖最近 90 天 305 个真实文件，所以**没有新的 type tag 需要加**，工作集中在字段级和目录元数据。

## 总体原则

**Pa.** 不改写原始字段语义。Rust 字段名严格遵循 camelCase ↔ snake_case 双向映射。当 JSON 含 `ID` / `UUID` 等末尾全大写片段（如 `sourceToolUseID`、`sourceToolAssistantUUID`）时，`#[serde(rename_all = "camelCase")]` 会把 `source_tool_use_id` 推断为 `sourceToolUseId`（小写 id），无法匹配，必须用显式 `#[serde(rename = "sourceToolUseID")]` 保留原拼写。

**Pb.** 分层零回归。每个 phase 独立可发布、可回滚；Phase 1 完成后 cc-session-jsonl 单独发版 0.2.0；Phase 2 依赖 Phase 1；Phase 3 仅消费 Phase 2 已经序列化好的 JSON。

**Pc.** 目录文件作"加速缓存"而非真相。`sessions-index.json` 若存在且新于目录 mtime，可直接读 entries；否则回退到全目录扫描。绝不在缺失 jsonl 但索引存在时返回幽灵 session。

**Pd.** 跨版本前向兼容。所有新字段 `Option<T>`；旧版本 JSONL 不会因为缺失新字段而失败。

## Phase 1 — cc-session-jsonl 0.2.0

### 1.1 字段补全（types/user.rs）

```rust
transcript_entry! {
    pub struct UserEntry {
        pub message: Option<UserContent>,
        pub is_meta: Option<bool>,
        pub permission_mode: Option<String>,
        pub tool_use_result: Option<serde_json::Value>,
        #[serde(rename = "sourceToolUseID")]
        pub source_tool_use_id: Option<String>,
        #[serde(rename = "sourceToolAssistantUUID")]
        pub source_tool_assistant_uuid: Option<String>,
    }
}
```

说明：
- `is_meta`：tool-use placeholder 标志。
- `permission_mode`：当前 turn 时的权限模式快照（与 `permission-mode` entry 区分：那是切换事件，这个是切换后第一条 user message 的 inline 标签）。
- `tool_use_result`：旧版 toolResult 回填（仍出现在最新 JSONL 中）。保留 `Value`，不在 parser 层规约结构。
- `source_tool_use_id` / `source_tool_assistant_uuid`：tool 调用与结果的关联 id。两个名字同时存在因为 Claude Code 自身在过渡命名。

### 1.2 字段补全（types/assistant.rs）

```rust
transcript_entry! {
    pub struct AssistantEntry {
        pub message: Option<ApiMessage>,
        pub request_id: Option<String>,
        // ... 既有字段 ...
        pub attribution_plugin: Option<String>,
        pub attribution_skill: Option<String>,
    }
}
```

说明：`attribution_plugin` / `attribution_skill` 是本次最高价值的归因维度。一个 assistant turn 可同时被打上 plugin 和 skill 标签（例：当前会话产生 64 条 `attributionSkill = "superpowers:brainstorming"`，`attributionPlugin = "superpowers"`）。

### 1.3 FileHistorySnapshot

**无改动**。当前已支持 `message_id`、`is_snapshot_update`、`snapshot`。

### 1.4 新模块 `manifest.rs`

新文件 `crates/cc-session-jsonl/src/manifest.rs`：

```rust
#[derive(Debug, Clone, Deserialize, Serialize)]
#[serde(rename_all = "camelCase")]
pub struct SessionsIndex {
    pub version: Option<u32>,
    pub original_path: Option<String>,
    pub entries: Option<Vec<SessionIndexEntry>>,
}

#[derive(Debug, Clone, Deserialize, Serialize)]
#[serde(rename_all = "camelCase")]
pub struct SessionIndexEntry {
    pub session_id: Option<String>,
    pub full_path: Option<String>,
    pub file_mtime: Option<i64>,
    pub first_prompt: Option<String>,
    pub message_count: Option<u64>,
    pub created: Option<String>,
    pub modified: Option<String>,
    pub git_branch: Option<String>,
    pub project_path: Option<String>,
    pub is_sidechain: Option<bool>,
    pub summary: Option<String>,
}

#[derive(Debug, Clone, Deserialize, Serialize)]
#[serde(rename_all = "camelCase")]
pub struct BridgePointer {
    pub session_id: Option<String>,
    pub environment_id: Option<String>,
    pub source: Option<String>,
}

pub fn read_sessions_index(project_dir: &Path) -> Option<SessionsIndex> { ... }
pub fn read_bridge_pointer(project_dir: &Path) -> Option<BridgePointer> { ... }
```

设计点：
- 顶层是独立模块（feature-gated 与 `scanner` 一同，复用 `cfg(feature = "scanner")`）。
- 返回 `Option<T>` 而不是 `Result<T, Err>`，因为「文件不存在」是常态而非错误。
- 不在这层做时间戳校验、缓存有效性判断，那是 scanner 层的事。

### 1.5 scanner.rs 扩展

`SessionFile` 增加两个字段：

```rust
pub struct SessionFile {
    // 既有字段保持不变
    pub session_id: String,
    pub project: Option<String>,
    pub path: PathBuf,
    pub is_agent: bool,
    pub parent_session_id: Option<String>,
    // 新增
    pub bridge: Option<manifest::BridgePointer>,
    pub index_entry: Option<manifest::SessionIndexEntry>,
}
```

`scan_sessions()` 行为：
1. 先按现有方式扫描 jsonl 文件（保持目录是真相）。
2. 每个 project 目录读一次 `sessions-index.json`，构建 `sessionId -> SessionIndexEntry` 映射，逐 file 填回 `index_entry`。索引里有但 jsonl 不在的条目**直接忽略**（防幽灵）。
3. 每个 project 目录读一次 `bridge-pointer.json`，**仅当**当前扫到的 session 的 `session_id` 与 bridge.session_id 匹配时，把 `bridge` 复制进对应 `SessionFile`。否则不传播（避免错误归因）。
4. `tool-results/` 目录已经隐式被跳过（scanner 只进 UUID 子目录找 `subagents/`），不需要改。

### 1.6 测试

新增测试：
- `parse_user_with_new_fields` — 全字段 round-trip。
- `parse_user_legacy_source_tool_assistant_uuid` — 旧名字段单独。
- `parse_assistant_with_attribution` — plugin/skill 同时出现。
- `parse_sessions_index_full` / `parse_sessions_index_minimal`。
- `parse_bridge_pointer`。
- `scan_with_sessions_index` — 索引与目录扫描合并。
- `scan_with_bridge_pointer` — bridge 仅挂到匹配 session。
- `scan_index_with_missing_jsonl` — 索引有但文件不存在的条目被忽略。

### 1.7 版本

`crates/cc-session-jsonl/Cargo.toml` → `version = "0.2.0"`（minor bump，新字段可选，向后兼容旧 0.1.x 的 JSONL）。

## Phase 2 — cc-token-usage 2.1.0

### 2.1 数据穿透

`src/data/types.rs` 中的 `ValidatedTurn` 与 `SessionData` 新增字段：

```rust
pub struct ValidatedTurn {
    // 既有
    pub attribution_plugin: Option<String>,
    pub attribution_skill: Option<String>,
    pub permission_mode: Option<String>,   // 来自 user entry inline
    pub is_meta: Option<bool>,
    pub source_tool_use_id: Option<String>,
}

pub struct SessionData {
    // 既有
    pub source: SessionSource,  // 新
    pub plugin_cost_breakdown: HashMap<String, AttributionStats>,  // 新
    pub skill_cost_breakdown: HashMap<String, AttributionStats>,   // 新
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub enum SessionSource {
    Local,
    ReplBridge { environment_id: Option<String> },
}

#[derive(Debug, Clone, Default, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct AttributionStats {
    pub cost: f64,
    pub turns: u64,
    pub input_tokens: u64,
    pub output_tokens: u64,
    pub cache_read_tokens: u64,
    pub cache_write_tokens: u64,
}
```

注意：JSON 字段名通过 `#[serde(rename_all = "camelCase")]` 保持与 JSONL 风格一致 — `attributionPlugin`、`attributionSkill`、`permissionMode`、`isMeta`，**不再造新名字**。

### 2.2 Pipeline 改动

`src/data/pipeline.rs`（或对应的 build_session_data 入口）：
1. 在 ValidatedTurn 构造时从 `AssistantEntry.attribution_plugin/skill` 与 `UserEntry.permission_mode/is_meta/source_tool_use_id` 抓字段。
2. 在 SessionData 聚合阶段，按 plugin 和 skill 各做一次 group-by，累加 `AttributionStats`。
3. 若 `SessionFile.bridge` 非空，设置 `SessionSource::ReplBridge`，否则 `Local`。

**Pipeline 不为此重排顺序**，attribution 仅作为只读 metadata 流过。

### 2.3 analysis 模块扩展

**overview.rs**：在现有 per-model、per-agent 输出后增加：
- `Plugin Cost Breakdown` 表
- `Skill Cost Breakdown` 表

排序：成本降序。空时不渲染（避免老数据出现空白卡片）。

**session.rs**：单 session 详情新增 `Source` 字段（Local / REPL Bridge）和 `Permission Mode` 切换序列（沿用既有 permission-mode entry 收集逻辑，但 inline 的 `permission_mode` 字段作为「快照标记」用于交叉验证，不重复计数）。

**validate.rs**：识别下列新 system.subtype 仅用于分类计数，不改变 token 计算逻辑（保留 dual-path 验证的独立性）：
- `away_summary` — Claude 离开窗口的总结
- `bridge_status` — Bridge 心跳/状态
- `scheduled_task_fire` — 计划任务触发
- `turn_duration` — turn 耗时元数据

这些 subtype **不参与** validate 的双路计数（既不计 tokens 也不算 api_error）— 它们对成本不构成贡献。

### 2.4 输出层

**json.rs / `HtmlReportPayload`**：增加
- `pluginCostBreakdown: { [name]: AttributionStats }`
- `skillCostBreakdown: { [name]: AttributionStats }`
- 每个 session 对象增加 `source: { kind: "local" | "replBridge", environmentId?: string }`

**text.rs**：在 overview 文本输出后加两个 comfy-table 块；列：Plugin/Skill、Cost、Turns、Input、Output。

**html.rs**：旧 fallback HTML 加两个 Chart.js 饼图，对应 plugin/skill 占比。（备用渲染，主路径走 Vue。）

### 2.5 测试

- pipeline 集成测试：构造含 attribution 的 ValidatedTurn fixture，断言 plugin/skill breakdown 正确累加。
- snapshot 测试：overview text 输出新增块的快照。
- validate 测试：新 subtype 不破坏既有计数。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 老 JSONL 缺新字段 | 全部 `Option<T>`；测试覆盖最小化 entry。 |
| `sessions-index.json` 与目录不同步 | 永远以目录为真相；索引仅作元数据补充。 |
| serde rename 漏掉 ID/UUID 大写 | 显式 `#[serde(rename)]`；测试用真实样例 round-trip。 |
| bridge-pointer 误挂到错 session | 严格按 `sessionId` 匹配，否则不挂。 |
| 前端 bundle 膨胀 | 不引入新依赖；复用现有 Chart.js。 |

## 不在本次范围

- 读取 `tool-results/<id>.txt`（巨型 tool 输出外置）— 留接口 `SessionFile.tool_results_dir`，下次按需做。
- 合并 plugin / skill / agent 成统一归因维度 — 它们是正交概念，强行合并会失真。
- REPL Bridge 远端 session 抓取 — 仅做来源标记，不去 Claude.ai 拉数据。
- 新的 system.subtype 引入 dual-path 计数 — 它们不产生 token 成本。

## 提交节奏

三个独立 commit：
1. `feat(jsonl): support v2.1 user/assistant attribution fields + project manifest files` — Phase 1。
2. `feat(analysis): plugin/skill cost breakdown + REPL bridge source tracking` — Phase 2。
3. `feat(frontend): plugin/skill cost cards + session source column` — Phase 3。

每个 phase 后均通过 cargo test + clippy 才进下一步。

## 团队工作流（每 Phase 必走）

按项目 `.claude/agents/` 中的 agent 编排，每个 Phase 内部按以下顺序流转，**严禁并行**（agent 间有依赖）：

1. **builder** — 按本 spec 实现 Phase N 的代码。仅写实现，不写测试。
2. **code-reviewer** — 对照本 spec、原始 cc_session_jsonl 类型、既有 pattern，审查 builder 产出。重点：原始字段语义是否被改写、`#[serde(rename)]` 是否到位、`Option<T>` 是否一致、是否新增超出 spec 的"顺手清理"。
3. **qa-engineer** — 写单元 + 集成测试。Phase 1 必须包含真实 fixture round-trip（从 `~/.claude/projects/` 抽真实 JSON line 作 test data，敏感字段脱敏）。Phase 2 必须覆盖空 attribution、单 plugin、多 plugin、plugin+skill 同时的 4 类场景。Phase 3 需要 npm build 后手测 + screenshot 描述。
4. **test-qa-lead** — 审查 qa-engineer 产出的测试覆盖度，识别遗漏（边界值、空集、并发等）。回流要求 qa-engineer 补测。
5. **diff-analyst** — **Phase 2 和 Phase 3 必走**：在 builder 改动前先对 `~/.claude/projects/` 跑 baseline（`cargo run -- --format json`），改完后再跑一次，逐字段对比；只允许新增字段、不允许既有字段值漂移（除非 spec 明确说"应该改"）。

**禁用并行**：上一个 agent 的报告是下一个 agent 的输入，并行会丢上下文。  
**自治循环**：每个 Phase 内部的 builder → code-reviewer → qa-engineer → test-qa-lead → diff-analyst → fix → commit 流程**不需要用户介入**，发现问题就回流到对应 agent 修复。只在以下时刻停下来等用户：

- spec 审阅前（当前节点）；
- 一个 Phase 完成 commit 之后（让用户看到提交记录）；
- 出现 spec 未覆盖的设计选择（必须升级给用户决策）。

**禁止伪发版**：每个 Phase commit 完成后**不自动**改 Cargo.toml 版本或打 tag。等三个 Phase 都完成、用户验收后再一并发版。
