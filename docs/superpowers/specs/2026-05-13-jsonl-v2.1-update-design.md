# JSONL v2.1+ 兼容性更新设计

**日期：** 2026-05-13
**范围：** parser (cc-session-jsonl) + 分析层 (cc-token-usage) + Vue 前端
**版本目标：** cc-session-jsonl 0.2.0（minor，向后兼容），**cc-token-usage 3.0.0**（major，JSON 输出含破坏性重组：`agentTurns` 改为嵌套 `subagents`）
**Phase 概览：** 三个 Phase 按数据流分层 — Phase 1 解析层（user / assistant / system 三类 entry 新字段统一补全）→ Phase 2 分析层（plugins/skills/hooks session-level 聚合 + agent_turns 重组为嵌套 subagents）→ Phase 3 展示层（前端 chip 行 + 重写 SessionEntry 接口适配新嵌套结构）。每一类工作只在它所属的 Phase 做一次。

## 核心认知

**项目的"钱"都来自 `assistant.message.usage`**：input_tokens、output_tokens、cache_creation/read。Claude Code 2.0.77+ 新增的字段（attribution_plugin / attribution_skill / is_meta / permission_mode / tool_use_result / hook_count / hook_infos 等）**不改变金额计算**，只是给"这次 turn 是谁触发的 / 触发了什么"加上标签。

所以本次更新的本质是：**让 parser 不丢字段，分析层把新字段当 session-level 元数据展示，不发明新的归因层级**。

## 数据模型（session-centric）

session 是顶层聚合根，所有数据围绕它组织：

```
session
 ├─ turns       (Vec<ValidatedTurn>)        ── 主 session 的 turns，派生：总 tokens / pricing / 活跃时间
 ├─ subagents   (Vec<Subagent>)     ★重组★  ── 每个 subagent 一个 entry，内嵌自己的 turns
 │   └─ each Subagent { agent_id, agent_type, description, turns: Vec<ValidatedTurn>, time_range }
 ├─ plugins     (Vec<PluginUsage>)  ★新★    ── 该 session 用了哪些 plugin (老版本就是空 Vec)
 ├─ skills      (Vec<SkillUsage>)   ★新★    ── 该 session 用了哪些 skill
 ├─ hooks       (Vec<HookUsage>)    ★新★    ── 该 session 用了哪些 hook
 └─ metadata    (SessionMetadata)           ── title / tags / mode / pr_links / 等
```

**`Vec<T>` 而非 `Option<Vec<T>>`**：本次 spec 不做跨 session 全局排名/趋势（违反 P0），所以"老版本没记录 vs 用户没用"在展示层不需要区分 — 都按"空 → 不渲染"处理。P0.5 字段稳定性意识保留为约束，但落在「未来若要做跨版本统计，必须先按版本分组」这条提醒上，不强制数据结构带 Option。

**关键变化**：
- 既有的扁平 `agent_turns: Vec<ValidatedTurn>` **被重组为嵌套** `subagents: Vec<Subagent>`，每个 Subagent 内嵌自己的 turns。语义对应「这个 session 召唤了哪几个 subagent，每个干了什么」。真实场景里单 session 已观察到 69 个 subagent，扁平结构无法表达。
- 新增三类子集合（plugins / skills / hooks）**与 turns / subagents 平级**，命名复数名词对称。
- 所有派生指标（总 tokens / pricing / 活跃时间 / 错误次数）**继续从 turns + subagents.turns 按需计算**，不在 SessionData 上加冗余字段。
- **subagent 文件实测仅含 user/assistant 两种 entry**（近 30 天：1132 assistant + 827 user，0 hook、0 attribution、0 metadata）。所以 `Subagent` 在本次只承载 turns + 时间范围 + meta 元信息；plugins / skills / hooks 三类聚合**只从 main session 的 turns / system entries 走**，不需要遍历 subagents。这条假设在 Phase 2 测试里要明确断言（未来如果格式变化能立刻发现）。

## 总体原则

**P0（最高原则）. Session-centric。** 所有数据最终都要落到「某个 session 用了什么 / 花了什么」的视角。不构造跨 session 全局排名、不构造 turn 维度的孤立分析、不构造去 session 化的聚合视图。判断任何新增维度是否合理的唯一标尺：**它能告诉用户"这个 session 是怎样的"吗？** 凡是只能输出"全部 session 上谁排第一"的指标，本次不做。

**P0.5（与 P0 平级）. 字段稳定性意识。** 不同字段引入 Claude Code 的时间不同：核心 cost 字段（`input_tokens`/`output_tokens`/`cache_*`/`model`）在 2.1.4+ 的全部 17038 个 turn 上 100% 稳定（A 级）；`sourceToolUseID` 从 2.1.71+ 出现（B 级）；`hookCount`/`permissionMode(inline)` 从 2.1.104+ 出现（C 级）；`attributionPlugin`/`attributionSkill` **只从 2.1.138+ 出现**（D 级，仅最近两个版本）。

设计后果：
1. **任何计算都不能让 D 级字段影响成本或 token 计数。** Cost / total tokens 永远只依赖 A 级字段，跨版本一致。
2. **本次只做 session detail 展示**，所以「数据不可知」与「确认没用」都按"空"展示 — 老 session 就是不显示 plugins/skills/hooks 行；不增加"灰字版本提示"等额外 UI。
3. **未来若引入跨 session plugin/skill/hook 趋势分析**（违反 P0，本次不做），必须先按版本分组 — 否则得到"plugin 使用率从 0% 暴涨到 100%"的伪结论。届时再把数据模型升级为 `Option<Vec<T>>` + `RecordedCapabilities`，不为本次提前预埋。
4. **任何新字段加入前都必须先实测最早出现版本**，写入下方版本边界表，否则不允许进 spec。

**Pa.** 不改写原始字段语义。Rust 字段名严格遵循 camelCase ↔ snake_case 双向映射。当 JSON 含 `ID` / `UUID` 等末尾全大写片段（如 `sourceToolUseID`、`sourceToolAssistantUUID`、`toolUseID`）时，`#[serde(rename_all = "camelCase")]` 会把 `source_tool_use_id` 推断为 `sourceToolUseId`（小写 id），无法匹配，必须用显式 `#[serde(rename = "sourceToolUseID")]` 保留原拼写。

**Pb.** 分层零回归。每个 phase 独立可发布、可回滚。

**Pc.** 跨版本前向兼容。所有新字段 `Option<T>`；旧版本 JSONL 不会因为缺失新字段而失败。

**Pd.** YAGNI。只补当前真实存在于数据里的字段，不为可能出现的字段预留位置。

**Pe.** 同层一次做完。增加 parser 字段属于"解析层"工作，统一在 Phase 1 做完；UserEntry、AssistantEntry、SystemEntry 的字段补全不分批。

## 明确排除（之前讨论过，决议不做）

- ❌ 解析 `sessions-index.json` — 数据不稳定，不可作权威源（D1）
- ❌ 解析 `bridge-pointer.json` — 非核心数据（D3）
- ❌ 读取 `tool-results/<id>.txt` 外置内容 — 对 token 计算无贡献；仅承认 `toolu_xxx.txt` 与 tool_use_id 的关联事实，不构造索引（D4）
- ❌ 识别新 `system.subtype` 值（away_summary、bridge_status、scheduled_task_fire、turn_duration）— 非核心数据，且不产生 token 成本（D5）
- ❌ 在 turn 维度交叉关联 plugin/skill 与 agent — 数据显示它们在同一 entry 上互斥，不强行拼接（D2 修正：作为 session-level 标签处理）

## Phase 1 — 解析层（cc-session-jsonl 0.2.0）

把 user / assistant / system 三类 entry 的新字段一次性补全。新字段全部 `Option<T>`，旧 JSONL 不受影响。

### 1.1 UserEntry 字段补全（types/user.rs）

```rust
transcript_entry! {
    pub struct UserEntry {
        pub message: Option<UserContent>,
        pub is_meta: Option<bool>,
        pub permission_mode: Option<String>,
        pub tool_use_result: Option<serde_json::Value>,
        #[serde(rename = "sourceToolUseID")]
        pub source_tool_use_id: Option<String>,
        #[serde(rename = "sourceToolAssistantUUID")]
        pub source_tool_assistant_uuid: Option<String>,
    }
}
```

| 字段 | 来源 JSON | 含义 |
|---|---|---|
| `is_meta` | `isMeta` | tool-use placeholder 标志 |
| `permission_mode` | `permissionMode` | 当前 turn 的权限模式快照（inline，与 `permission-mode` 切换事件 entry 不同） |
| `tool_use_result` | `toolUseResult` | 旧版 toolResult 回填，保留 `Value`，parser 层不规约结构 |
| `source_tool_use_id` | `sourceToolUseID` | tool 调用与结果的关联 id（新名） |
| `source_tool_assistant_uuid` | `sourceToolAssistantUUID` | tool 调用与结果的关联 id（旧名，仍在用） |

### 1.2 AssistantEntry 字段补全（types/assistant.rs）

```rust
transcript_entry! {
    pub struct AssistantEntry {
        // ... 既有字段保持不变 ...
        pub attribution_plugin: Option<String>,
        pub attribution_skill: Option<String>,
    }
}
```

| 字段 | 来源 JSON | 含义 |
|---|---|---|
| `attribution_plugin` | `attributionPlugin` | 触发本 turn 的 plugin 名（如 `superpowers`） |
| `attribution_skill` | `attributionSkill` | 触发本 turn 的 skill 名（如 `superpowers:brainstorming`） |

### 1.3 SystemEntry 字段补全（types/user.rs，SystemEntry 当前在该模块）

Hooks 在 JSONL 里以 `system` entry 的 `subtype = "stop_hook_summary"` 形式记录，**结构化字段放在 entry 顶层而非 message 内**。当前 `SystemEntry` 只声明 `message` / `subtype` / `duration_ms`，导致 hook 相关 7 个顶层字段被 serde 静默丢弃（近 90 天数据有 624 条 hook 触发记录）。

真实样例：
```json
{
  "type": "system", "subtype": "stop_hook_summary",
  "hookCount": 1,
  "hookInfos": [{"command": "bash ${CLAUDE_PLUGIN_ROOT}/hooks/emit-event.sh", "durationMs": 20}],
  "hookErrors": [], "preventedContinuation": false,
  "level": "suggestion", "toolUseID": "e7953fc8-..."
}
```

补全：

```rust
transcript_entry! {
    pub struct SystemEntry {
        pub message: Option<serde_json::Value>,
        pub subtype: Option<String>,
        pub duration_ms: Option<u64>,
        // ── 仅在 subtype = stop_hook_summary 时出现 ──
        pub hook_count: Option<u64>,
        pub hook_infos: Option<Vec<HookInfo>>,
        pub hook_errors: Option<Vec<serde_json::Value>>,
        pub prevented_continuation: Option<bool>,
        pub stop_reason: Option<String>,
        pub has_output: Option<bool>,
        pub level: Option<String>,
        #[serde(rename = "toolUseID")]
        pub tool_use_id: Option<String>,
    }
}

#[derive(Debug, Clone, Deserialize, Serialize)]
#[serde(rename_all = "camelCase")]
pub struct HookInfo {
    pub command: Option<String>,
    pub duration_ms: Option<u64>,
}
```

注意：`toolUseID` 末尾全大写「ID」必须显式 `#[serde(rename = "toolUseID")]`。`hook_errors` 用 `Vec<Value>` 兜底，因为错误结构未必稳定。

### 1.4 测试（三层防御，区分"能解析"与"解析正确"）

`Option<T>` 只保证不 panic，**不能告诉你字段值是否对得上**。下面三层缺一不可。

#### Layer A — Round-trip 值断言（不只断 is_some）

每个新字段一个测试，用真实采样作 fixture，**值要逐项对齐**：

- `parse_user_with_is_meta_and_source_tool_use_id` — 断言 `entry.is_meta == Some(false)` 且 `entry.source_tool_use_id.as_deref() == Some("toolu_01XXXX")`。
- `parse_user_with_source_tool_assistant_uuid_legacy` — 断言两个 ID 字段独立解析。
- `parse_user_with_permission_mode_inline` — 断言 `permission_mode.as_deref() == Some("bypassPermissions")`，且**不**误归类到 `PermissionModeEntry`。
- `parse_user_with_tool_use_result` — `Value::Object` 任意嵌套都能 round-trip。
- `parse_assistant_with_attribution_plugin_and_skill` — 断言 `attribution_plugin.as_deref() == Some("superpowers")` 且 `attribution_skill.as_deref() == Some("superpowers:brainstorming")`。
- `parse_assistant_legacy_no_attribution` — 旧 JSONL 全字段 `None`。
- `parse_system_stop_hook_summary` — 断言 `hook_count == Some(1)`、`hook_infos[0].command.as_deref() == Some("bash ${CLAUDE_PLUGIN_ROOT}/hooks/emit-event.sh")`、`hook_infos[0].duration_ms == Some(20)`、`prevented_continuation == Some(false)`、`tool_use_id.as_deref() == Some("e7953fc8-...")`。
- `parse_system_stop_hook_summary_with_errors` — 非空 `hook_errors`，逐项断言数组内容。

#### Layer B — 字段填充率快照对账

**真实风险：** `#[serde(rename = "sourceToolUseID")]` 写错一个字母，所有测试可能仍 pass（只要测试只构造一个字段就肯定能 round-trip 自己），但跑真实数据时该字段永远 `None`，悄无声息丢失数据。

防护测试：对每个新字段，比较「JSONL 文件里 grep 该原始 JSON key 的命中数」与「parser 解析后 `Some(_)` 的命中数」。两者必须相等。

```rust
#[test]
fn fill_rate_attribution_plugin() {
    let fixture = include_str!("../tests/fixtures/real_sessions/attribution_sample.jsonl");
    let raw_grep = fixture.lines().filter(|l| l.contains("\"attributionPlugin\"")).count();
    let parsed = fixture.lines()
        .filter_map(|l| serde_json::from_str::<Entry>(l).ok())
        .filter(|e| matches!(e, Entry::Assistant(a) if a.attribution_plugin.is_some()))
        .count();
    assert_eq!(raw_grep, parsed,
        "字段填充率不一致: JSONL 里有 {} 条 attributionPlugin，parser 只解析出 {} 条",
        raw_grep, parsed);
}
```

对所有 11 个新字段做同样断言：`is_meta` / `permission_mode` / `tool_use_result` / `source_tool_use_id` / `source_tool_assistant_uuid` / `attribution_plugin` / `attribution_skill` / `hook_count` / `hook_infos` / `prevented_continuation` / `tool_use_id`（SystemEntry 上）。

#### Layer C — 历史回归（既有字段零漂移）

新字段加进结构体后，**既有字段的解析必须完全不变**。Phase 1 完成后：
- 跑 `cargo run -p cc-token-usage -- validate` — 必须 PASS 全部 115+ session（与当前 1076/1076 一致）。
- 跑 `cargo run -p cc-token-usage -- --format json > new.json`，与改动前的 `old.json` diff，**所有既有字段值逐位对齐**（用 `jq` 排除 `subagents` / `plugins` / `skills` / `hooks` 等新字段，比对剩下的应当 byte-equal）。

### 1.5 scanner / metadata 模块

**无改动**。`sessions-index.json` / `bridge-pointer.json` / `tool-results/` 都不读取。现有 scanner 的 3 种目录布局支持已覆盖最新数据，`tool-results/` 因为不在 scanner 进入的 UUID 子目录的 `subagents/` 路径下，已经隐式被忽略。

### 1.6 版本

`crates/cc-session-jsonl/Cargo.toml` → `version = "0.2.0"`。Minor bump，向后兼容所有 0.1.x 时期的 JSONL（新字段都 `Option`）。

## Phase 2 — 分析层（cc-token-usage 3.0.0，major bump）

把 Phase 1 暴露的新字段聚合为 session-level metadata（plugins / skills / hooks），**并把扁平的 `agent_turns` 重组为嵌套的 `subagents`**。后者是破坏性数据结构变更，触发 major 版本。

### 2.1 数据穿透与重组

`src/data/models.rs`（注意当前模块叫 `models.rs` 不是 `types.rs`）：

### 2.2 Pipeline 改动

`src/data/pipeline.rs`：
1. **subagent 重组**：把当前所有进 `agent_turns: Vec<ValidatedTurn>` 的代码路径，改为按 `agent_id` 分组进 `Vec<Subagent>`；同时读取 `cc_session_jsonl::scanner::load_agent_meta` 拿到 `.meta.json` 的 `agentType` / `description` 注入到 `Subagent.agent_type` / `.description`。`first_timestamp` / `last_timestamp` 由该 subagent 自己的 turns 推导。
2. **plugin / skill 聚合**：ValidatedTurn 构造时从 `AssistantEntry.attribution_plugin / attribution_skill` 抓字段；SessionData 聚合阶段**仅遍历 `turns`**（主 session），按 plugin / skill 字符串分组累加 `turns / cost / tokens`，结果写入 `SessionData.plugins` / `.skills`。老 session 因为源 JSONL 没字段，所有 turn 的 attribution_plugin 都是 None，聚合结果自然是空 Vec。**不**遍历 subagents.turns（实测 subagent 文件无 attribution；加断言锁死，未来格式变化能立刻被发现）。
3. **hooks 聚合**：仅遍历 **main session** 的 SystemEntry where `subtype == "stop_hook_summary"`（subagent 文件实测没有 system entries），对每个 `hookInfos[].command` 累加：`invocations += 1`、`total_duration_ms += durationMs`、`error_count += if hookErrors 非空 1 else 0`、`prevented_continuation_count += if preventedContinuation 1 else 0`。结果写入 `SessionData.hooks`。
4. **派生指标继续按需算**：总 tokens / pricing / 活跃时间 / 错误次数等，**不**在 SessionData 上加冗余字段。`total_turn_count()` 等 method 改为 `self.turns.len() + self.subagents.iter().map(|s| s.turns.len()).sum::<usize>()`。
5. **不**做 plugin → agent 反向追溯。**不**把 plugin/skill 混入 per-subagent breakdown。
6. **不引入 semver 依赖**。本次不区分「数据不可知 vs 用户没用」— 都按"空 Vec"处理。版本边界表保留在本 spec 作为未来扩展时的参考数据。

### 2.4 输出层

**text.rs（终端）**：在 session detail 输出末尾，**非空时**才渲染对应行（与既有 metadata 风格一致）：

```
Subagents: general-purpose (12 turns, $0.18, "深度调研陈伟星") | code (4 turns, $0.05, "写测试")
Plugins:   superpowers (64 turns, $0.42)
Skills:    superpowers:brainstorming (64 turns, $0.42)
Hooks:     bash ${CLAUDE_PLUGIN_ROOT}/hooks/emit-event.sh (3 invocations, 60 ms total)
```

老 session（无 subagents / plugins / skills / hooks 数据）输出就是不渲染这几行，完全不出现版本提示。**Overview 命令不增加新表**——避免跨 session 全局排名。

**json.rs / `HtmlReportPayload`**：每个 session 对象的 JSON 输出**破坏性重组**：
- 删除：`agentTurns: [...]`（旧扁平字段）
- 新增：`subagents: [{ agentId, agentType, description, turns: [...], firstTimestamp, lastTimestamp }, ...]`
- 新增：`plugins`、`skills`、`hooks` 三个数组（与 `turns` / `subagents` 同级，老 session 是 `[]`）

**html.rs（fallback）**：不动。

### 2.5 validate.rs

**无改动**。dual-path 计数器仍只看 `usage` 字段；attribution / hooks 字段进入 metadata 通道，不参与金额校验。

### 2.6 测试（聚合不变量必须显式断言）

- pipeline 单测（subagents）：构造 main + 多 subagent 文件夹场景，断言 `subagents` 按 agent_id 分组正确、每个 Subagent.turns 数量与原始 agent-X.jsonl 一致、`.meta.json` 的 agent_type / description 被注入。
- pipeline 单测（多 subagent）：使用真实采样（如 69 subagent 那个 session 的 fixture）验证内存数据结构正确。
- pipeline 单测（plugins/skills）：构造含 attribution 的 ValidatedTurn 序列，断言 `plugins` / `skills` 聚合正确。
- pipeline 单测（hooks）：同 command 多次出现 → invocations / total_duration_ms / error_count / prevented_continuation_count 正确累加。
- 迁移回归：所有依赖 `agent_turns` 的旧测试改造为新嵌套结构后，**聚合数字（如 total_agent_turn_count）保持一致**。
- 边界：空 subagents / 空 attribution / 空 hooks / 单 plugin / 单 skill / 多 hook / hook 含错误 / preventedContinuation=true。
- json snapshot：新字段命名 / 顺序稳定；旧 `agentTurns` 字段确实从输出中消失。

**聚合不变量（必须显式断言）**：

```rust
// 重组无损：所有 subagent turn 加起来必须 = 原扁平 agent_turns 长度
assert_eq!(
    session.subagents.iter().map(|s| s.turns.len()).sum::<usize>(),
    legacy_agent_turn_count,
    "agent_turns -> subagents 重组丢失 turn"
);

// 聚合不漏不重：plugins 总 turn 计数 = main session 带 attribution 的 turn 计数
assert_eq!(
    session.plugins.iter().map(|p| p.turns).sum::<u64>(),
    session.turns.iter().filter(|t| t.attribution_plugin.is_some()).count() as u64,
);

// 上限约束：plugin cost ≤ session 总 cost（因为不是每个 turn 都有 plugin）
assert!(
    session.plugins.iter().map(|p| p.cost).sum::<f64>()
        <= session.turns.iter().map(turn_cost).sum::<f64>() + EPSILON,
);

// hooks 聚合无漏：invocations 总数 = sum(SystemEntry.hookCount where subtype=stop_hook_summary)
assert_eq!(
    session.hooks.iter().map(|h| h.invocations).sum::<u64>(),
    raw_hook_count_from_system_entries,
);

// 假设回归：subagent.turns 上不应该有 attribution（实测假设）
for sub in &session.subagents {
    for turn in &sub.turns {
        assert!(turn.attribution_plugin.is_none(),
            "subagent 上出现了 attributionPlugin，与实测假设不符，需重新设计");
        assert!(turn.attribution_skill.is_none(), ...);
    }
}
```

### 3.1 类型契约

`frontend/src/types.ts`：

```ts
interface SessionEntry {
  // 既有 (turns, ...)
  // agentTurns?: ValidatedTurn[];     // 删除：本次重组，改为 subagents
  subagents?: Subagent[];
  plugins?: PluginUsage[];               // 老版本就是空数组 [] 或省略

  hooks?: HookUsage[];
}

interface Subagent {
  agentId: string;
  agentType?: string;
  description?: string;
  turns: ValidatedTurn[];
  firstTimestamp?: string;
  lastTimestamp?: string;
}

interface PluginUsage { plugin: string; turns: number; cost: number; inputTokens: number; outputTokens: number; }
interface SkillUsage  { skill: string;  turns: number; cost: number; inputTokens: number; outputTokens: number; }
interface HookUsage   { command: string; invocations: number; totalDurationMs: number; errorCount: number; preventedContinuationCount: number; }
```

## 团队工作流（每 Phase 必走）

按 `.claude/agents/` 内的 agent 编排，**严禁并行**（agent 间有依赖）：

1. **builder** — 按本 spec 实现 Phase N 代码。仅写实现，不写测试。
2. **code-reviewer** — 对照本 spec、原始 cc_session_jsonl 类型、既有 pattern，审查 builder 产出。重点：原始字段名拼写是否完整保留、`#[serde(rename)]` 是否到位、`Option<T>` 是否一致、是否新增超出 spec 的"顺手清理"。
3. **qa-engineer** — 写单元 + 集成测试。Phase 1 必须包含从 `~/.claude/projects/` 抽真实 JSON line 作 test data 的 round-trip。
4. **test-qa-lead** — 审查 qa-engineer 产出的测试覆盖度，识别遗漏。回流要求 qa-engineer 补测。
5. **diff-analyst** — **Phase 2 和 Phase 3 必走**：在改动前先对 `~/.claude/projects/` 跑 baseline（`cargo run -- --format json` 存盘），改完后再跑，逐字段对比。除了本 spec 明确说新增的 `plugins` / `skills` / `hooks` / `attributionPlugin` / `attributionSkill` 等字段外，**既有字段值不允许漂移**。

**自治循环**：每 Phase 内 builder → code-reviewer → qa-engineer → test-qa-lead → diff-analyst → fix → commit **不需要用户介入**，发现问题回流到对应 agent 修复。只在以下时刻停下来：
- spec 审阅前（当前节点）；
- 一个 Phase commit 完成后；
- 出现 spec 未覆盖的设计选择必须升级。

**禁止伪发版**：每个 Phase commit 完成后**不自动**改 Cargo.toml 版本或打 tag。等三个 Phase 都完成、用户验收后再一并发版。
